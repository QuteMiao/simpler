fanin_count = fanin_count + 1 设计

💡 为什么要 +1 冗余？

在任务提交时，生产者可能已经完成（如链式依赖中前一个任务执行很快）。如果不加这个冗余，可能出现：

  1. 生产者已完成，但消费者还没加入fanout链表
  2. 导致生产者永远收不到消费者的引用，fanout_refcount 无法达到 fanout_count
  3. 缓冲区无法被正确释放

+1 冗余 = 初始化时立即释放一个引用（相当于 scope 持有的引用），确保即使没有消费者加入链表，任务也能正确完成生命周期。

// pto_orchestrator.cpp, 行 390-427 // STEP 5: Fanout 构建 - 早期完成优化 int32_t early_finished = 0; task.fanin_count = fanin_count + 1; // +1 冗余设计 payload->fanin_actual_count = fanin_count; // 实际数量（不含+1） for (int i = 0; i < fanin_count; i++) { PTO2TaskDescriptor& producer = ...; pto2_fanout_lock(producer); // 获取生产者锁 int32_t prod_state = sched->task_state[prod_slot].load(acquire); if (prod_state >= PTO2_TASK_COMPLETED) { // 早期完成优化：生产者已完工，直接减少依赖计数 early_finished++; } else { // 正常路径：加入fanout链表 producer.fanout_head = orch->dep_pool_cur_entry; } producer.fanout_count += 1; // 增加生产者引用 pto2_fanout_unlock(producer); } // 批量初始化：合并早期完成 + scope持有的+1 引用 int32_t initial_refcount = early_finished + 1; int32_t new_rc = sched->fanin_refcount[slot].fetch_add(initial_refcount, acq_rel) + initial_refcount; // 检查是否就绪（fanin_count 包含 +1） if (new_rc >= fanin_count + 1) { sched->ready_queues[task.worker_type].push(task_id); }

五、fanout_lock 自旋锁机制

fanout_lock 用于保护 fanout_head 和 fanout_count 的并发访问。因为调度器可能在任务完成时遍历fanout链表，而orchestrator可能在提交新消费者时修改同一个链表。

🔒 锁的作用域
✓ 写入时加锁：orchestrator 添加消费者到 fanout 链表时
✓ 读取时加锁：scheduler 遍历 fanout 链表时
// pto_runtime2_types.h, 行 410-425 // 加锁实现：TAS (Test-And-Set) 自旋锁 static inline void pto2_fanout_lock(PTO2TaskDescriptor& task) { for (;;) { // 1. 轮询检查锁是否被释放 while (task.fanout_lock.load(acquire) != 0) { SPIN_WAIT_HINT(); // CPU让出等待 } // 2. 尝试CAS获取锁 int32_t expected = 0; if (task.fanout_lock.compare_exchange_weak(expected, 1, acquire, relaxed)) { return; // 获取锁成功 } // 3. CAS失败，继续轮询 } } // 解锁实现 static inline void pto2_fanout_unlock(PTO2TaskDescriptor& task) { task.fanout_lock.store(0, release); // 释放锁 }

⚠️ 关键：锁内操作必须轻量

设计原则

fanout_lock 是任务级别的细粒度锁。在锁内只能执行：

  • fanout_head 指针的读取/写入
  • fanout_count 的增减
  • DepListEntry 节点的插入

禁止在锁内执行可能导致阻塞的操作（如内存分配、系统调用）。

七、完整生命周期示例

STEP 1 任务提交 (pto2_submit_task)
  • fanout_count = 1（scope持有）
  • fanout_head = nullptr
  • fanin_count = 依赖数量 + 1
  • fanin_refcount = 0
STEP 2 后续消费者提交
  • 获取 producer.fanout_lock
  • fanout_count += 1
  • fanout_head = 新节点（头插）
  • 释放 fanout_lock
STEP 3 任务完成 (on_task_complete)
  • task_state → COMPLETED
  • 获取 fanout_lock
  • 读取 fanout_head
  • 释放 fanout_lock
  • 遍历fanout链表，调用 release_fanin_and_check_ready
STEP 4 消费者引用释放 (release_producer)
  • fanout_refcount += 1（原子操作）
  • 检查 fanout_refcount == fanout_count
  • 如果相等，task_state → CONSUMED
  • 尝试推进 ring pointer（释放缓冲区）
STEP 5 Scope 结束 (pto2_scope_end)
  • 遍历 scope 中的所有任务
  • 对每个任务调用 release_producer
  • fanout_refcount += 1（释放scope持有的引用）
  • 触发 CONSUMED 状态转换（如果有消费者）

八、总结表格

字段 类型 初始化位置 修改位置 用途
fanin_count int32_t（静态） pto2_submit_task 无（只读） 记录任务依赖的生产者数量
fanin_refcount atomic<int32> 0 release_producer / fetch_add 已完成的依赖计数，=fanin_count时就绪
fanout_count int32_t（静态） pto2_submit_task (=1) pto2_submit_task (+=消费者) 需要的消费者释放计数
fanout_refcount atomic<int32> 0 release_producer / fetch_add 已释放的消费者引用 =fanout_count时释放缓冲区
fanout_head PTO2DepListEntry* nullptr pto2_submit_task（头插） 消费者链表头指针
fanout_lock atomic<int32> 0 CAS (0→1) / store(0) fanout链表并发访问保护
🎯 关键设计要点
1 +1 冗余设计：fanin_count = 实际依赖 + 1，避免早期完成导致的引用泄漏
2 早期完成优化：生产者已完成时，直接增加计数而不加入fanout链表
3 Scope 持有引用：每个任务的 fanout_count 初始为 1，表示创建它的 scope 持有引用
4 细粒度锁：fanout_lock 只保护 fanout_head/fanout_count 的访问，遍历时先复制指针再解锁
5 冷热分离：fanin_tasks 放在 TaskPayload（冷路径），减少 TaskDescriptor（热路径）大小
