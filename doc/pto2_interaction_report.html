PTO2 任务提交与调度交互分析

pto2_submit_task 与 AicpuExecutor::resolve_and_dispatch_pto2 交互逻辑深度解析

一、概述

PTO2（Process Tensor Orchestration 2）是用于异构计算平台的任务调度系统。该系统由两个核心组件构成：

系统架构概览
◆ Orchestrator（协调器）: 负责任务图构建、依赖分析、缓冲区分配
◆ Scheduler（调度器）: 负责任务分发、状态管理、核心分配

两个组件通过共享内存（Shared Memory）进行通信，实现无锁的并发任务处理。

二、核心函数定位

pto2_submit_task
文件: pto_orchestrator.cpp, 行号: 221

在 Orchestrator 线程中执行，负责构建任务依赖图、分配缓冲区、注册任务到调度器。

AicpuExecutor::resolve_and_dispatch_pto2
文件: aicpu_executor.cpp, 行号: 778

在 AICPU 调度线程中执行，负责检查任务完成状态、分发就绪任务到空闲核心。

六、resolve_and_dispatch_pto2 详细流程

Phase 0
初始化与等待
获取共享内存指针，初始化性能统计，进行一次性初始化（性能缓冲区分配）
Phase 1
检查任务完成
检查运行中的AIC/AIV核心是否有任务完成，调用 check_running_cores_for_completion
Phase 2
处理完成通知
对于完成的任务，遍历fanout链表，通知所有消费者任务，通过 release_fanin_and_check_ready 减少依赖计数
Phase 3
分发就绪任务
从ReadyQueue中取出就绪任务，分发到空闲的AIC/AIV核心，调用 dispatch_ready_tasks_to_idle_cores
Phase 4
处理延迟释放
批量处理延迟的fanin释放，调用 on_task_release 遍历fanin列表释放生产者
Phase 5
检查完成条件
如果所有任务完成（completed_tasks >= total_tasks），退出调度循环

八、任务状态转换

状态 名称 描述 转换条件
PENDING 等待中 等待依赖完成 fanin_refcount < fanin_count
READY 就绪 所有依赖已满足，等待调度 fanin_refcount == fanin_count
RUNNING 运行中 正在核心上执行 被调度到AIC/AIV核心
COMPLETED 已完成 核心执行完成，输出可用 硬件核心报告完成
CONSUMED 已消费 所有消费者已完成，缓冲区可释放 fanout_refcount == fanout_count

九、ReadyQueue 实现

PTO2使用无锁的MPMC（多生产者多消费者）队列，基于 Dmitry Vyukov 的设计：

PTO2ReadyQueue 关键特性
enqueue_pos atomic<uint64> 生产者位置（独立缓存行）
dequeue_pos atomic<uint64> 消费者位置（独立缓存行）
slots[] ReadyQueueSlot[] 每槽包含sequence计数器防止ABA问题

十、关键代码路径

pto2_submit_task 核心代码路径

// 1. 分配任务槽位
int32_t task_id = task_ring.pto2_task_ring_alloc();

// 2. 查找输入依赖
orch->tensor_map.lookup(payload->tensors[i], lookup_result);

// 3. 注册输出
orch->tensor_map.insert(payload->tensors[i], task_id, is_output);

// 4. 构建fanout链表（带自旋锁）
pto2_fanout_lock(producer);
producer.fanout_head = orch->dep_pool_cur_entry;
pto2_fanout_unlock(producer);

// 5. 检查是否就绪并加入队列
if (new_rc >= fanin_count + 1) {
    sched->ready_queues[task.worker_type].push(task_id);
}

resolve_and_dispatch_pto2 核心代码路径

// 1. 检查运行中的核心是否完成
check_running_cores_for_completion<CoreType::AIC>(...);

// 2. 通知消费者
while (current != nullptr) {
    release_fanin_and_check_ready(consumer_id, consumer);
    current = current->next;
}

// 3. 从就绪队列取出任务
int32_t task_id = ready_queues[CUBE].pop();

// 4. 分发到空闲核心

十一、性能优化要点

1. 无锁设计
调度器使用原子操作和内存序，避免使用锁。fanout链表使用轻量级自旋锁。
2. 缓存行对齐
ReadyQueue的enqueue_pos和dequeue_pos分别位于不同缓存行，避免伪共享。
3. 批量处理
空闲时批量处理延迟的fanin释放，减少锁竞争。
4. 早期就绪优化
如果生产者已完成的消费者，直接减少依赖计数，避免加入fanout链表。
5. 延迟释放
不在任务完成时立即释放生产者，延迟到空闲时批量处理。
6. 数据局部性
TaskPayload与TaskDescriptor分离，保证热路径（完成检查）的缓存效率。

十二、总结

pto2_submit_task 与 resolve_and_dispatch_pto2 的关系
🔗 生产者-消费者模式: pto2_submit_task 构建任务依赖图并提交任务，resolve_and_dispatch_pto2 消费就绪任务并执行
📤 共享内存通信: 通过 TaskDescriptor、TaskPayload 和 ReadyQueue 进行无锁数据交换
⏱️ 流水线并行: Orchestrator 可以持续提交任务，Scheduler 并行执行，形成流水线
🔄 依赖传播: Scheduler 在任务完成时通过 fanout 链表传播完成信号，触发消费者就绪
